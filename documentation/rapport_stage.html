1. Introduction

Dans le cadre de ma formation en Développement Digital – Full Stack à l'OFPPT, j'ai effectué un stage de fin de formation d'une durée de 5 semaines au sein de l'entreprise Promptis, spécialisée dans le consulting IT à Casablanca, de janvier à février 2026.

Ce stage m'a été confié avec un objectif clair : concevoir et développer un système complet répondant à un besoin réel de l'entreprise. Le projet, baptisé Promptis Manager, est une application web full-stack de gestion des ressources humaines et portail client.

Avant ce stage, Promptis gérait ses informations de manière dispersée : fichiers Excel, e-mails et outils hétérogènes. L'objectif était de centraliser la gestion des employés, des clients, des projets et des tickets de support dans une interface unique, moderne et sécurisée.

Ce rapport retrace les différentes étapes du projet, depuis l'analyse des besoins jusqu'à la livraison technique, en passant par la conception architecturale et les choix technologiques effectués.

3. Contexte et Problématique

3.1 Contexte du Projet

En tant qu'entreprise de consulting IT, Promptis gère quotidiennement un ensemble d'entités interdépendantes : des équipes d'employés, un portefeuille de clients, des projets en cours et un flux constant de demandes de support. Jusqu'au lancement de ce projet, ces informations étaient gérées de manière fragmentée — fichiers Excel, e-mails et outils non intégrés — rendant le suivi difficile, chronophage et propice aux erreurs.

3.2 Problématique

Comment centraliser et optimiser la gestion des ressources humaines, des projets clients et du support technique dans un outil unique, accessible et sécurisé ?

Principaux défis identifiés :

  • Absence de vision globale sur l'activité de l'entreprise
  • Difficulté à suivre les affectations et compétences des employés sur les projets
  • Manque de centralisation des livrables et fichiers par projet
  • Gestion des tickets de support non structurée, sans priorités ni traçabilité
  • Besoin de contrôle d'accès granulaire selon les rôles (admin, employé, client)

5. Choix Technologiques & Architecture

5.1 Stack technique Globale

La conception de Promptis Manager repose sur une architecture moderne de type Single Page Application (SPA) en isolant parfaitement le front-end du back-end. Cette séparation garantit une flexibilité technique optimale, des performances accrues et une maintenabilité simplifiée.

Approche adoptée : API RESTful (Backend) + Client SPA réactif (Frontend)

5.2 Justification des choix Architecturaux

Une étape fondamentale d'ingénierie consistait à justifier nos choix face à d'autres alternatives populaires sur le marché. Aucun framework n'a été choisi par hasard ; ils répondent à des problématiques précises du cahier des charges de Promptis.

1. Pourquoi React au lieu de Vue.js ou Angular ?

Bien que Vue.js soit natifs à l'écosystème Laravel (via Inertia.js), le choix s'est porté sur React en raison de son immense écosystème de librairies tierces (notamment Recharts pour nos tableaux de bord) et de son modèle de réconciliation via le Virtual DOM, crucial pour une SPA manipulant des centaines de variables dynamiques (tickets, états d'avancement). De plus, React favorise une approche 100% découplée de l'API.

2. Pourquoi Laravel au lieu de Node.js (Express/Nest) ?

Laravel (PHP 8) a été présélectionné pour sa maturité prodigieuse sur les aspects backend conventionnels. Son ORM Eloquent gère les relations de bases de données avec une syntaxe hyper-expressive, et son écosystème natif de sécurité embarque d'office la protection contre les injections SQL, XSS, et inclut une validation de requêtes très stricte (FormRequests). Pour une plateforme B2B manipulant des données clients sensibles, Laravel garantit une fiabilité immédiate.

3. Pourquoi Sanctum au lieu de JWT (JSON Web Tokens) ?

Bien que JWT soit un standard, son implémentation implique une gestion complexe de la révocation des tokens. Laravel Sanctum fonctionne sur un principe de Personal Access Tokens stockés en base, permettant à un administrateur Promptis de révoquer immédiatement l'accès d'un compte compromis d'un simple clic. C'est une obligation sécuritaire primordiale dans notre contexte PME.

4. Pourquoi Tailwind CSS au lieu de Bootstrap ?

Bootstrap impose un langage de design très reconnaissable et lourd. Tailwind permet de construire l'esthétique exacte du Soft-UI souhaitée, sans jamais quitter le fichier du composant React. Le compilateur de Tailwind (JIT) garantit que seuls les styles réellement utilisés seront expédiés en production (CSS < 20ko).

5.3 Architecture Applicative Détaillée

L'application utilise un modèle en couches (Layered Architecture). Le cycle de vie d'une donnée, depuis l'interaction de l'utilisateur jusqu'à sa sauvegarde en base, suit un cheminement immuable garantissant la traçabilité des opérations.

Flux Applicatif Global (Data Flow)

[ USER BROWSER ]
↓ (1) Clic & interaction React State
[ REACT CONTEXT + COMPONENT ]
↓ (2) Appel HTTP Axios Asynchrone (avec Token Sanctum)
[ ROUTES API LARAVEL (api.php) ]
↓ (3) Filtrage par Middlewares (Auth & RBAC)
[ FORM REQUEST (Validation de données) ]
↓ (4) Acheminement validé
[ CONTROLLER PHP ]
↓ (5) Logique Métier & Interaction ORM
[ ELOQUENT ORM MODELS ]
↓ (6) Requêtes SQL Protégées
[ MYSQL DATABASE ]

Figure 7 – Cycle de communication Client / Serveur / BDD

Cette approche apporte deux avantages majeurs :

  • Isolement des erreurs : Si l'API renvoie une 422 Unprocessable Entity (ex: email invalide), React capture l'erreur via Axios et la traduit en interface visuelle, sans que l'application globale ne s'interrompe.
  • Sécurité des injections : Le contrôleur MVC n'accepte jamais les requêtes "Raw". Les StoreRequest de Laravel assainissent systématiquement chaque paramètre (strip_tags, type hints).

5.4 Architecture de Sécurité (RBAC & Tokens)

Parce que Promptis Manager contient des documents stratégiques B2B, une architecture RBAC (Role-Based Access Control) a été implémentée au cœur du système sur trois axes : L'interface, le routeur API, et la logique intra-entité.

Le Modèle de Permissions strict

Module Administrateur (Promptis) Employé Client Externe
Projets CRUD complet (tous) Lecture seule (assignés uniquement) Lecture seule (ses propres projets)
Tickets CRUD complet, assignation Modification d'état, réponse Création, réponse
Employés/Clients CRUD complet, gestion MDP Accès refusé Accès refusé
Fichiers Upload / Download complet Upload / Download (sur assignation) Download uniquement
Triple Verrouillage Sécuritaire :
  • Frontend Verrou : Utilisation de Contextes React (useAuth) pour cacher purement et simplement les menus et boutons selon la valeur de user.role.
  • Routing Verrou : Côté Laravel, les routes comme /api/clients sont protégées par le middleware admin. Un Postman avec un token d'employé se verra opposer une 403 Forbidden.
  • Model Scopes Verrou : Même s'ils ont accès à /api/projects, un Global Scope Eloquent force la BDD à ne renvoyer que les enregistrements rattachés à Auth::id() via les tables pivots.

6.2 Design System & Expérience Utilisateur (Soft-UI)

Une attention exceptionnelle a été portée à l'esthétique et à l'ergonomie (UX/UI). Les interfaces B2B sont souvent froides et génériques. Pour Promptis Manager, le but était de délivrer un produit "Premium", qui valorise l'image de marque de Promptis face à ses propres clients.

La charte "Soft-UI" : plutôt que d'utiliser des bordures dures et des couleurs d'alerte primaires, nous avons implémenté un système de design moderne axé sur :
1. Des ombres diffuses (drop-shadow-sm) remplaçant les séparations nettes.
2. Une palette HSL cohérente : variations de bleu océan (Primary) contrastées par du blanc pur et un fond de page #f8fafc (Slate 50).
3. Des micro-animations : "Hover states" sur tous les tableaux, translations subtiles (-2px) sur les boutons, renvoyant un feedback immédiat à l'utilisateur.

Composants réutilisables (React)

Pour maintenir une cohérence et une vélocité de rendu sur les 8 écrans majeurs, nous avons codé nos propres composants primitifs UI. Par exemple, au lieu de réécrire les statistiques de haut de page dans chaque module, un composant <StatCard value={...} title={...} icon={...} trend={...} /> a été créé. Ce pattern DRY (Don't Repeat Yourself) divise la taille du code final par deux.

L'iconographie repose entièrement sur la librairie HeroIcons (SVG pur), garantissant une netteté parfaite des vecteurs sans le poids d'une librairie icon-font complète.

6.3 Module 1 : Authentification & Inscription

Ce module agit comme la porte d'entrée inviolable de l'application. La sécurité y est omni-présente, cachée sous une interface attrayante.

Figure 8 - Vues des pages de Connexion et d'Inscription

Détail technique de l'implémentation

Lors d'un Login, React envoie les credentials (email, password) via Axios POST. Laravel vérifie la base de données. En cas de succès, Sanctum génère un token cryptographique de 40 caractères.

Côté React, ce token est stocké dans le localStorage, tandis que les informations du profil (rôle, nom) sont injectées dans un état global Context API. Dès lors, chaque requête HTTP sortante passée par l'application inclut automatiquement ce token grâce à un Intercepteur Axios configuré dans src/config/axios.js :

axiosClient.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = "Bearer " + token;
  return config;
});

6.4 Module 2 : Tableau de Bord & Statistiques dynamiques

Le Dashboard est l'écran directionnel d'atterrissage. Son but est d'offrir en un clin d'œil la santé globale de l'organisation pour l'administrateur, ou un récapitulatif personnel pour le client.

Figure 9 - Tableau de bord côté administrateur

Détail technique de l'implémentation

Pour éviter l'accumulation de dizaines de requêtes API (N+1 query problem), le backend Laravel a été optimisé pour compiler l'intégralité des KPIs (chiffre d'affaires brut, tickets en cours, taux de progression moyen) en une unique requête SQL agrégée au sein de DashboardController.

Le Frontend réceptionne alors une payload JSON parfaitement structurée et mappe les données dans la librairie Recharts. Nous avons implémenté des "Sparklines" en SVG pur avec un dégradé (Gradient) dynamique sous les courbes pour offrir cet aspect Soft-UI de haute qualité. Le rendu graphique s'adapte sans scintillement au redimensionnement de l'écran.

6.5 Module 3 : Gestion B2B (Employés & Clients)

L'administration opérationnelle implique la gestion stricte du personnel interne (employés Promptis) et des entreprises externes (clients du portail).

Figure 10 - Interface de gestion Clients et Employés

Détail technique de l'implémentation

Sur cette interface, l'accent a été mis sur la gestion unifiée des formulaires. Plutôt que de développer une page distincte pour créer un élément et une autre pour l'éditer, un composant Modale superposé (Overlay React) est exploité dans les deux cas.

La validation bénéficie d'une approche hybride : une pré-validation en JavaScript (vérifiant instantanément la longueur ou le pattern d'email) empêchant les envois inutiles. Ensuite, Laravel intervient via ses FormRequests, garantissant par exemple que l'adresse email d'un nouvel employé est unique dans le système avec la règle stricte 'email' => 'required|email|unique:users'.

6.6 Module 4 : Gestion des Projets

Le module projets est le cœur métier de l'outil. Il relie ensemble N employés à 1 client sur une période donnée et gère l'état d'avancement ainsi que le budget alloué par Promptis.

Figure 11 - Module Projets et état métier

Détail technique de l'implémentation

Techniquement, les projets exploitent une relation complexe Many-to-Many entre Project et User. Du côté Laravel Eloquent, l'enregistrement se fait en cascade : on crée le projet, la liaison au client (Foreign Key), puis la synchronisation de l'équipe via la méthode $project->employees()->sync($request->employees) qui garantit un pivot exact.

UI-wise, l'appli offre un filtrage avancé. En tapant dans une barre de recherche spécifique native à la vue, la fonction .filter() de JavaScript recalcule le tableau affiché en millisecondes, offrant une recherche temps-réel instantanée (Zero-latency filtering) avant même de relancer une recherche serveur.

6.7 Module 5 : Tickets de Support

Les tickets assurent l'ITSM (IT Service Management). Ce module remplace les échanges par email erratiques par un log centralisé, horodaté et commentable lié directement à l'ID d'un projet.

Figure 12 - Système de Ticketing Helpdesk

Détail technique de l'implémentation

Le statut des tickets ("Ouvert", "En cours", "Résolu") exploite un mapping colorimétrique automatisé sur le front (Rouge, Orange, Vert). Le point de force de ce module réside dans sa relation One-to-Many Polymorphique simulée via les logs (Comments). La vue détaillée d'un ticket déclenche un useEffect secondaire en React qui Fetch tout l'historique asynchrone des ticket_comments, générant une chronologie style "Chat" très fluide.

6.8 Module 6 : Gestion des Fichiers

Un GED (Gestionnaire Électronique de Documents) léger conçu pour rendre le transfert manuel obsolète. Les clients téléversent des spécifications, l'équipe uploade les livrables .zip et docs de rendu.

Figure 13 - UI/UX du GED intégré

Détail technique de l'implémentation

L'upload implique du protocole Multipart. Frontend via React : on instancie un nouvel objet FormData(), dans lequel on ajoute le binaire file via un input de type file natif stylisé.

Backend via Laravel : le contrôleur réceptionne et utilise la Facade Storage::disk('public')->put(). Les fichiers ne sont pas conservés en BDD de manière binaire, seule le chemin physique (Path URI) l'est (ex: storage/files/doc123.pdf), maximisant ainsi les IO/s et réduisant le gonflage MySQL. Le Frontend lit l'extension (ex: .pdf, .docx, .png) et affiche dynamiquement l'icône HeroIcon adéquate pour rendre l'interface intuitive.

6.9 Outils Transverses : Recherche Globale & Notifications

Certains outils ne sont pas liés à une page précise mais gravitent tout autour de l'application : ce sont nos fonctionnalités d'accessibilité rapide.

6.9.1 La Recherche Globale Universelle

Intégrée nativement en haut de la Sidebar persistance, la barre de recherche permet à un administrateur Promptis de taper "Alpha" et de recevoir immédiatement dans un panneau déroulant le Projet Alpha, le Client Société Alpha, ou tout Ticket contenant Alpha. Sur le backend, un endpoint API global invoque trois classes de modèle distinctes en parallèle, consolidées dans une collection unifiée JSON via Eloquent.

6.9.2 Polling de Notifications

Plutôt que de forcer un rechargement F5, l'application exécute de manière invisible un setInterval JavaScript pour invoquer l'API Laravel chaque minute, et s'informer des modifications passives (tickets assignés). Un indicateur cloche devient rouge vif. Ce mécanisme, moins coûteux en infrastructures qu'un serveur WebSockets (Pusher), répond parfaitement aux besoins synchrones du cabinet de consulting.

7. Difficultés Rencontrées et Solutions

Mon stage formatif n'a pas été sans obstacles techniques professionnels. Les résoudre a constitué l'essentiel de mon apprentissage architectural pratique.

Difficulté Technique majeure Impact sur le projet Analyse logicielle & Solution Ingénierie
CORS Policy Blocked : Lors de la communication React/Laravel. Bloquage intégral des appels de connexion d'environnement. Configuration complète du fichier cors.php de Laravel origin. Ajout du support credentials.
Perte d'authentification après F5 Déconnexion frustrante d'utilisateur lors du rechargement de vue. Mise en place d'un contexte React useAuth() et stockage du Token via localStorage. Hook useEffect au mount.
Problèmes N+1 Queries (Eloquent BDD) Temps de charge des Dashboards s'effondrant (lenteurs API). Mise en application du Eager Loading syntaxe Laravel afin de réduire de 100 requêtes SQL à seulement 2 requêtes indexées.
Rendu React StrictMode Duplication involontaire d'appels API lors des tests de développement. Nettoyage du code asynchrone dans des fonctions de cleanup() du useEffect, garantissant l'Idempotence SPA.
Erreurs JSON Parsing (API 500) L'application plantait au lieu d'afficher une alerte (Crash blanc). Attrapage via bloc try catch coté Axios, conversion de l'objet erreur en messages flash toast élégants.

8. Exploitation, Tests et Déploiement

8.1 Stratégie de tests

L'assurance qualité a été gérée manuellement et rigoureusement via Postman pour la couche applicative Backend, et visuellement en navigateur pour le Frontend.

  • Tests Unitaires BDD : Les routes de création ont été poussées aux limites avec envoi de Formulaires vides pour s'assurer que PHP/Laravel renvoyait systématiquement les JSON d'erreurs de validation standardisés.
  • Tests d'Intégration d'Etat (React) : Vérification constante de la désactivation (disable) du bouton de soumission le temps que l'API réponde, évitant la duplication de clics frénétiques (Double-submission bug).
  • Test de Sécurité RBAC : Tentative d'accès depuis le Token d'un ID Client à la route de destruction (DEL) d'un profil Employé /api/employees/delete. La réponse vérifiée en 403 Action Unauthorized a validé le mur de sécurité.

8.2 Qualité de code & Conventions

Une stricte hygiène de code a été imposée afin de maintenir la lisibilité par d'autres ingénieurs de Promptis :

  • PascalCase pour la nomenclature des composants React (ex: StatCard.jsx).
  • camelCase pour la définition des variables JavaScript.
  • snake_case imposé pour absolument tous les noms des colonnes MYSQL.
  • Factorisation intégrale des appels API à l'intérieur d'un Service de Config.

8.3 Préparation au déploiement production

Pour le passage en mode production du SI au terme du stage :

1. Backend : Exécution de php artisan optimize et stockage des configurations en fichiers plats via config:cache. Basculement de l'environnement DEV à PROD.
2. Frontend : Compilation par Vite via npm run build générant des fichiers HTML et JS minimisé, transpilés, hashés garantissant la révocation de cache navigateur lors des MAJ.
